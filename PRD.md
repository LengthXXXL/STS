本 PRD 是后续开发的主指令文档。开发时请优先阅读本节和对应阶段，不要凭空扩展范围。

## 1. 产品定位

STS 是一个面向普通股票用户的模拟交易与策略回测网站。用户不需要写代码，可以通过选择“策略积木插件”组合出简单量化交易策略，并使用按需获取的历史行情数据进行回测。

系统还提供论坛，让用户分享策略、策略积木插件、回测结果和交易想法。

一句话目标：

```text
让不懂量化编程的股票用户，也能像搭积木一样创建、回测、分享简单交易策略。
```

## 2. MVP 总目标

第一版 MVP 必须跑通以下主链路：

```text
注册/登录
  -> 查看或创建策略积木插件
  -> 用插件创建策略
  -> 选择策略并配置回测区间
  -> 系统按需获取该策略需要的行情数据
  -> 执行回测
  -> 查看回测结果
  -> 分享策略或回测结果到论坛
  -> 首页展示热门推荐
```

MVP 必须覆盖：

- 增删改查
- 模糊查询
- 分页
- JWT 登录鉴权
- 前后端分离
- 文件上传下载
- 角色权限
- 简单推荐算法
- 策略积木插件
- 策略构建
- 按需行情获取
- 历史模拟回测
- 论坛分享

MVP 暂不覆盖：

- 实盘交易
- 券商账户接入
- 高频交易
- 分钟级回测
- Tick 数据
- 全市场历史行情本地存储
- 复杂撮合引擎
- 复杂拖拽工作流
- 机器学习策略生成
- 多资产类型交易

## 3. 关键约束：行情数据不能全量本地化

由于本地电脑内存和存储有限，STS 不设计成“把全部股市历史数据导入 MySQL”的系统。

### 3.1 正确的数据策略

用户发起回测时，系统按以下流程获取行情：

```text
读取策略配置
  -> 解析股票池插件
  -> 得到需要的股票代码列表
  -> 读取用户选择的开始日期和结束日期
  -> 查询本地轻量缓存是否已有可用数据
  -> 缺失部分通过数据源适配器按需获取
  -> 回测引擎使用本次所需数据计算
  -> 保存回测结果，不保存全市场历史数据
```

### 3.2 本地数据库可以保存什么

可以保存：

- 用户、角色、权限。
- 策略积木插件。
- 策略配置。
- 论坛内容。
- 文件记录。
- 回测任务、回测结果、交易明细、收益曲线。
- 少量按需行情缓存。
- 缓存元数据，例如股票代码、日期范围、数据源、过期时间。

不应该保存：

- 全市场所有股票的完整历史行情。
- 用户没有回测过的无关股票历史数据。
- 无期限增长的大规模行情缓存。

### 3.3 行情缓存原则

MVP 建议保留轻量缓存，减少重复请求，但缓存必须可控：

- 缓存只保存用户回测实际请求过的数据。
- 缓存表需要按股票代码和日期建立索引。
- 缓存需要支持过期时间。
- 管理员可以清理缓存。
- 后续可改为文件缓存或对象存储，不强依赖 MySQL。

### 3.4 数据源适配器原则

行情获取必须通过统一接口封装：

```text
MarketDataProvider.fetch_daily(symbols, start_date, end_date)
```

第一版可以先实现一个简单数据源：

- 用户上传小范围 CSV。
- 或接入一个公开行情数据源。
- 或实现一个可替换的爬取适配器。

无论使用哪种方式，回测引擎都只依赖统一的行情数据格式，不直接依赖具体爬虫或第三方 API。

## 9. 阶段 3：策略积木插件

阶段目标：实现策略构建的基础积木。此阶段重点是定义插件、管理插件、初始化内置插件，不要求执行完整回测。

### 9.1 插件分类

策略积木插件分为 6 类：

- 股票池插件
- 指标插件
- 条件插件
- 买入动作插件
- 卖出动作插件
- 风控插件

### 9.2 插件表字段

建议字段：

- `id`
- `name`
- `code`
- `category`
- `description`
- `params_schema`
- `default_params`
- `logic_key`
- `visibility`
- `status`
- `creator_id`
- `created_at`
- `updated_at`

字段说明：

- `code`：插件唯一编码，例如 `indicator.ma`。
- `category`：插件分类。
- `params_schema`：前端动态渲染表单用的参数定义 JSON。
- `default_params`：默认参数 JSON。
- `logic_key`：后端执行回测时识别计算逻辑。
- `visibility`：`private` 或 `public`。
- `status`：`active`、`disabled`、`deleted`。

### 9.3 第一批必须实现的插件

第一批插件必须能支撑一个“均线交叉策略”完整运行。

股票池插件：

- `stock_pool.single_stock`：单股票，用户输入一个股票代码。
- `stock_pool.stock_list`：股票列表，用户输入多个股票代码。

指标插件：

- `indicator.close_price`：收盘价。
- `indicator.volume`：成交量。
- `indicator.ma`：N 日简单移动平均线。

条件插件：

- `condition.gt`：A > B。
- `condition.lt`：A < B。
- `condition.cross_up`：A 上穿 B。
- `condition.cross_down`：A 下穿 B。

买入动作插件：

- `action.buy_fixed_amount`：固定金额买入。
- `action.buy_fixed_ratio`：按可用现金比例买入。

卖出动作插件：

- `action.sell_all`：全部卖出。
- `action.sell_fixed_ratio`：固定比例卖出。

风控插件：

- `risk.stop_loss`：固定比例止损。
- `risk.take_profit`：固定比例止盈。
- `risk.max_position`：单股最大仓位限制。

## 12. 阶段 6：简化回测引擎

阶段目标：基于用户选择的策略，按需获取行情数据，执行简化日线回测并保存结果。

### 12.1 MVP 回测假设

- 使用日线数据。
- 买入价使用收盘价。
- 卖出价使用收盘价。
- 不考虑滑点。
- 不考虑涨跌停无法成交。
- 不考虑停牌。
- 不考虑分红复权。
- 不支持融资融券。
- 不支持 T+0。
- 手续费可配置，默认 0.1%。

### 12.2 回测主流程

```text
读取策略配置
  -> 解析股票池得到 symbols
  -> 读取回测开始日期和结束日期
  -> 通过 MarketDataProvider 获取所需日线数据
  -> 计算指标
  -> 逐交易日判断卖出条件和风控
  -> 执行卖出
  -> 判断买入条件
  -> 执行买入
  -> 更新现金、持仓、总资产
  -> 记录每日资产
  -> 计算收益和风险指标
  -> 保存回测结果
```

```text
  -> 系统只获取 000001 在该区间所需日线数据
  -> 执行回测
  -> 查看收益曲线和交易明细
```

验收重点：

- 不抓取无关股票。
- 不导入全市场历史行情。
- 交易明细和资金曲线可查看。

### 19.2 管理员管理缓存

```text
管理员登录
  -> 进入行情缓存管理
  -> 查看已缓存的股票和日期范围
  -> 清理过期缓存
```

验收重点：

- 缓存可查询。
- 缓存可清理。
- 清理缓存不影响已保存的回测结果。

### 19.3 用户分享策略

```text
用户打开策略详情
  -> 设置策略公开
  -> 进入论坛发帖
  -> 关联公开策略
  -> 发布帖子
  -> 其他用户浏览帖子
  -> 其他用户复制策略
```

验收重点：

- 私有策略不能被他人查看。
- 公开策略可以被复制。

### 19.4 首页推荐

```text
用户点赞、收藏、复制策略
  -> 系统记录行为日志
  -> 推荐接口按规则计算热度
  -> 首页展示热门策略和热门帖子
```

验收重点：

- 推荐内容只展示公开资源。
- 推荐结果会随用户行为变化。

## 21. 默认实现决策

以下内容作为本 PRD 的默认决策，后续开发直接按此执行。除非用户明确提出变更，否则不要在实现过程中反复询问。

1. 后端使用 FastAPI。
2. 前端使用 Vue 3 + Vite + TypeScript + Element Plus。
3. 状态管理使用 Pinia。
4. 图表展示使用 ECharts。
5. 数据库使用 MySQL 8.0。
6. 第一版策略构建使用表单式，不做拖拽式编排。
7. 第一版回测使用简化日线成交规则。
8. 第一版行情数据采用“CSV 小范围上传 + 按需缓存”作为可落地数据源。
9. 行情数据服务必须保留 `MarketDataProvider` 适配器接口，后续可追加公开数据源或爬虫适配器。
10. 第一版不实现全市场历史行情本地入库。
11. 第一版创建默认管理员账号初始化逻辑，账号信息通过环境变量配置。
12. 第一版论坛内容发布后直接展示，管理员可事后删除或下架，暂不做发布前审核。
13. 第一版推荐算法使用规则推荐，不使用机器学习。
14. 第一版文件存储使用本地文件目录，后续可追加对象存储。
15. 第一版回测任务可以同步执行，后续再追加异步任务队列。

如果后续需要改变上述默认决策，只能在用户明确允许后修改；否则只能追加新的扩展任务。
